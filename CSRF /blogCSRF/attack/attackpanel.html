What just happened?

After clicking Like in the attacker panel, a hidden cross-site request was sent using your session cookie. The blog accepted it and auto-created a post. Status changes from Idle → Attacked when the blog iframe reloads.

Signal: user click → victim iframe reload → status flip.

Why did the attack succeed?
  • No CSRF token validation on the post-creation endpoint.
  • Browser auto-attached the session cookie (credentials: "include").
  • No Origin/Referer checks; SameSite not enforced.
View vulnerable handler snippet 
// blog.php (simplified)
if (!empty($_POST['content'])) {
  // No CSRF token check
  // No Origin/Referer validation
  save_post($_SESSION['user_id'], $_POST['content']);
}

This endpoint trusts cookies blindly — classic CSRF vulnerability.

View attacker code 
// attacker_defense.html (what the attacker tries to do)
  function launchAttack(){
    const fd = new FormData();
    fd.append('content', 'CSRF attack attempt (no token)'); // no secret token added
    fetch('post.php', {
      method: 'POST',
      credentials: 'include', // sends your cookies automatically
      body: fd                 // but still missing the secret token
    }).then(async res => {
      const txt = await res.text().catch(()=>'');
      alert(res.status + ' — ' + (res.ok ? 'OK' : ('Blocked: ' + txt)));
    }).catch(e => alert('Fetch error: ' + e));
  }
  • It forges a submit: the attacker builds a POST that looks like “you clicked”.
  • Cookies ride along: the browser will attach your cookies when credentials:'include' is set, so the request seems authenticated.
  • But no secret token: the attacker cannot read or inject the victim page’s hidden csrf_token.
  • Server refuses: the token check fails, the server returns 403, and the blog remains unchanged.

Note: the attacker can send cookies, but cookies alone are not sufficient — the server requires the session token that only the genuine page can embed and send.
